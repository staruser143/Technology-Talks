Short answer: **no, business users shouldn't be created as individual IAM users** — the recommended path is SSO through AWS IAM Identity Center, not direct IAM identities.

**The recommended pattern — IAM Identity Center (SSO)**
The Amazon DataZone data portal is designed to be used principally by AWS IAM Identity Center–authenticated users, and it's actually a first-party AWS Identity Center application for SSO users. Practically, that means:

- Identity Center can use its own built-in directory, or you connect an external IdP (Okta, Entra ID, etc.) via SAML/SCIM, so users authenticate with credentials they already use elsewhere by setting up automatic provisioning of users and groups in IAM Identity Center so that users and groups in the external IdP domain are synced in.
- When you enable Identity Center for a DataZone domain, you choose between two assignment modes: with implicit user assignment, any user added to your IAM Identity Center directory can access your Amazon DataZone domain, while with explicit user assignment, you add specific users or groups from your Identity Center directory to grant access. Most orgs use explicit assignment, adding groups (e.g. "Finance Analysts," "Marketing Team") rather than individuals one by one.
- This can be scoped to a single AWS account rather than a whole Organization — DataZone administrators can enable IAM Identity Center for a single AWS account instead of the entire AWS organization, useful if a business unit doesn't have org-level access.

**Can IAM users/roles be used instead?** Technically yes, but it's the exception, not the default. The data portal authenticates users with IAM credentials or existing credentials from your identity provider through AWS IAM Identity Center, and you can attach a small policy (just `datazone:GetIamPortalLoginUrl`) to an IAM user/role/group to let them federate in. But there's a real limitation: only IAM principals in the domain's own AWS account can access that domain's data portal — IAM principals from other AWS accounts cannot. That makes the pure-IAM route impractical for most business-user populations, who usually live outside the data platform account entirely. This path tends to get used for service accounts, admins, or programmatic API/SDK access rather than end-user browsing.

**What happens once they're in.** Regardless of how they authenticated, DataZone maps the authenticated context to a user profile, associates that profile with domains and projects, and stores the permission state in an association table — user profile permissions to use projects and create entities are managed through domain units and policy grants, while within a specific project, the membership designation (owner, contributor, viewer) determines what actions are authorized. So the SSO login just gets them *into* DataZone; project membership and the subscription workflow we diagrammed earlier is what actually governs what they can see and request.

One more useful detail for the "how does this connect to the earlier diagrams" thread: when a user runs a query, their data portal client requests IAM session credentials that DataZone generates based on their effective profile in that project context, scoped to both the user's permissions and the project's resources — dropping them into Athena or Redshift with all the underlying IAM work abstracted away. So the IAM roles from the Lake Formation mapping diagram are entirely DataZone-managed service roles — the business user themselves never touches IAM directly.
