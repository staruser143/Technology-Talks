For dynamic layouts/content in Freemarker without managing numerous templates, consider these approaches:

1. Layout Fragments with Includes

Create reusable fragments and assemble dynamically:

```ftl
<#assign layoutType = model.layoutType!"default">

<#if layoutType == "twoColumn">
  <#include "layouts/twoColumn.ftl">
<#elseif layoutType == "grid">
  <#include "layouts/grid.ftl">
<#else>
  <#include "layouts/default.ftl">
</#if>

<#-- Inside layout files, include components dynamically -->
<#include "${componentName}.ftl">
```

2. Component Registry Pattern

Create a component registry in data model:

```java
// In your controller
Map<String, String> components = Map.of(
  "header", model.getHeaderType(),
  "body", model.getBodyComponent(),
  "footer", model.getFooterType()
);
model.addAttribute("components", components);
```

```ftl
<#list components?keys as key>
  <#include "components/${key}/${components[key]}.ftl">
</#list>
```

3. Meta-Template with Directives

Use macros for dynamic structure:

```ftl
<#macro renderField field>
  <#switch field.type>
    <#case "text">
      <input type="text" name="${field.name}" value="${field.value!}">
      <#break>
    <#case "select">
      <select name="${field.name}">
        <#list field.options as option>
          <option value="${option.value}">${option.label}</option>
        </#list>
      </select>
      <#break>
    <#case "custom">
      <#-- Evaluate custom template string -->
      <@field.customTemplate?interpret />
      <#break>
  </#switch>
</#macro>

<#list model.fields as field>
  <@renderField field />
</#list>
```

4. Template Composition with Configuration Objects

Pass layout configuration in JSON format:

```ftl
<#-- model.layoutConfig = {"rows": [{"columns": [{"component": "userCard", "width": 6}, ...]}]} -->

<#list model.layoutConfig.rows as row>
  <div class="row">
    <#list row.columns as col>
      <div class="col-md-${col.width}">
        <#if col.component?has_content>
          <#include "components/${col.component}.ftl">
        <#elseif col.content?has_content>
          ${col.content}
        </#if>
      </div>
    </#list>
  </div>
</#list>
```

5. Template String Evaluation (Use with caution)

For maximum flexibility but less security:

```ftl
<#-- model.dynamicTemplate contains template as string -->
<#attempt>
  <#assign templateSource = model.dynamicTemplate>
  <#assign ftlTemplate = templateSource?interpret>
  <@ftlTemplate />
<#recover>
  <#include "fallback.ftl">
</#attempt>
```

6. Hybrid Approach with Layout DSL

Create a simple DSL in your model:

```java
// Define layout structure
Layout layout = new Layout()
  .withSection("header", "banner")
  .withSection("content", new Grid()
    .withColumn(6, "profileCard")
    .withColumn(6, "statsPanel"))
  .withSection("footer", "standardFooter");
```

```ftl
<#list layout.sections as section>
  <section id="${section.name}">
    <#if section.type == "component">
      <#include "components/${section.value}.ftl">
    <#elseif section.type == "grid">
      <div class="row">
        <#list section.columns as col>
          <div class="col-md-${col.width}">
            <#include "components/${col.component}.ftl">
          </div>
        </#list>
      </div>
    </#if>
  </section>
</#list>
```

Best Practices:

1. Template Caching: Cache compiled templates when using dynamic includes
2. Security: Validate template paths to prevent directory traversal
3. Fallbacks: Always provide default templates
4. Configuration: Externalize layout definitions in JSON/YAML files
5. Performance: Use ?has_content checks before includes

Recommended Architecture:

```
templates/
├── layouts/
│   ├── base.ftl           # Base structure
│   ├── twoColumn.ftl
│   └── grid.ftl
├── components/
│   ├── header/
│   │   ├── default.ftl
│   │   └── banner.ftl
│   └── cards/
│       └── profile.ftl
└── dynamic.ftl           # Main orchestrator
```

The most maintainable approach is usually #2 (Component Registry) or #4 (Configuration Objects), as they separate layout definition from template logic while maintaining type safety and clear structure.
